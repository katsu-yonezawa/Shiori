# Shiori 技術方針

## 目的

Shiori は、自分のためのノートアプリとして小さく始める。最初から多機能なサービスを目指すのではなく、日常的に安心して書けること、後から端末を増やせること、運用コストを抑えられることを優先する。

初期版では macOS クライアントを主対象にする。将来的には iPhone と Web でも使えるようにするため、UI、ドメインロジック、同期処理をできるだけ分離しておく。

## 基本方針

- ローカルファーストで作る
- まずは macOS で快適に動くことを優先する
- クラウド同期はシンプルな差分同期から始める
- 本文中心のノートを想定し、添付ファイルは後から拡張する
- 無料枠または低コストで運用できる構成にする
- 将来の iPhone / Web 展開を妨げない構造にする

ローカルファーストとは、普段の読み書きを端末内のデータベースに対して行い、クラウドとは裏側で同期する考え方を指す。ネットワークが不安定でも書けることを重視し、クラウドはバックアップと複数端末同期のために使う。

## 採用予定の技術構成

| 領域 | 方針 |
| --- | --- |
| macOS アプリ | Tauri |
| UI | React / TypeScript |
| ローカルDB | SQLite |
| 全文検索 | SQLite FTS5 |
| クラウドDB | Supabase Postgres |
| 認証 | Supabase Auth |
| 同期 | 自前の差分同期から開始 |
| 添付ファイル | 初期版では最小限。必要に応じて Cloudflare R2 を検討 |

Tauri を使う理由は、Web 技術で UI を作りながら、macOS 向けに軽いデスクトップアプリを構成しやすいため。Electron よりアプリサイズやメモリ使用量を抑えやすく、将来的なモバイル対応の余地も残せる。

SQLite は、ノートアプリのようなローカル保存中心のアプリと相性がよい。ノート、タグ、削除状態、更新日時、同期状態を素直に管理でき、全文検索も FTS5 で実装できる。

Supabase は、Postgres、認証、Row Level Security、ストレージをまとめて扱えるため、個人開発の初期クラウド基盤として扱いやすい。AWS は将来必要な部分だけ使う方針とし、初期版では採用しない。

## 想定ディレクトリ構成

```text
apps/
  desktop/
    Tauri + React
  web/
    将来追加する Web 版

packages/
  core/
    ノート、タグ、検索、同期キューなどの共通ロジック
  ui/
    共通UIコンポーネント
  editor/
    エディタ関連
  schema/
    DBスキーマ、型定義、マイグレーション
```

初期段階では `apps/desktop` を中心に開発する。将来 Web 版や iPhone 版を作る時に再利用したい処理は、早めに `packages/core` へ寄せる。

## データ設計の初期方針

最初に扱う主なデータは、ノート、タグ、ノートとタグの関連、同期イベントとする。

```text
notes
- id
- user_id
- title
- body
- body_format
- created_at
- updated_at
- deleted_at
- version
- device_id
- sync_status
```

削除はすぐに物理削除せず、`deleted_at` を入れて削除済みとして扱う。複数端末で同期する場合、削除情報も同期対象にしないと、別端末からノートが復活する可能性があるため。

`version` と `updated_at` は競合検出に使う。`device_id` は、どの端末から変更されたかを追跡するために持たせる。

## 同期方針

初期版の同期は、ローカルの変更をキューに積み、一定間隔でクラウドへ送る方式にする。

```text
アプリ操作
  -> SQLiteへ即時保存
  -> sync_eventsへ変更を記録
  -> バックグラウンドでSupabaseへ送信
  -> サーバー側の変更をpull
  -> SQLiteへ反映
```

本文を1文字入力するたびにクラウドへ送らない。数秒の debounce、ノートを離れた時、アプリ終了時、一定間隔などでまとめて同期する。これにより、通信量と競合の発生を抑える。

競合解決は、最初は安全性を優先する。同じノートが複数端末で更新されていた場合、片方を黙って上書きせず、競合コピーとして残す。

```text
元のノート
元のノート（競合コピー）
```

リアルタイム共同編集は初期範囲に含めない。必要になった時点で、Yjs などの CRDT を検討する。

## コスト方針

初期版は、ほぼ無料で運用できる構成を目指す。

本文中心の個人ノートであれば、Supabase Free の範囲に収まる可能性が高い。コストが増えやすいのは、画像、PDF、音声、動画などの添付ファイルである。

添付ファイルが増えてきた場合は、次のように分ける。

| データ | 保存先 |
| --- | --- |
| ノート本文、タグ、同期情報 | Supabase Postgres |
| 画像、PDFなどの添付 | Cloudflare R2 または Supabase Storage |

当面は本文中心で設計し、添付ファイルは後から拡張する。

## 初期スコープ

最初の版では、次の機能に絞る。

- ノートの作成、編集、削除
- Markdown またはプレーンテキストでの本文編集
- タイトル、本文、更新日時の保存
- ローカル全文検索
- タグ付け
- Supabase へのログイン
- 手動同期または簡易的な自動同期
- 競合時のコピー保存

次の機能は、初期版では対象外とする。

- リアルタイム共同編集
- 複雑な権限管理
- 公開共有リンク
- 高度な添付ファイル管理
- AI要約やAI検索
- 業務向けの監査ログ

## 段階的な進め方

1. macOS でローカル保存できる最小版を作る
2. SQLite のスキーマとマイグレーションを整える
3. 検索とタグを入れる
4. Supabase Auth と Postgres をつなぐ
5. 差分同期を実装する
6. 競合時にコピーを残す
7. Web 版の実現性を確認する
8. iPhone 版の方式を改めて選ぶ

## 保留している判断

| 論点 | 初期判断 |
| --- | --- |
| エディタ | Markdown 中心なら CodeMirror 6、リッチテキスト重視なら TipTap を検討 |
| iPhone版 | Tauri mobile または React Native / Expo を比較して決める |
| 添付ファイル | 初期版では最小限。増えたら Cloudflare R2 を検討 |
| 同期エンジン | まず自前同期。複雑化したら PowerSync などを検討 |
| AWS | 初期版では使わない。S3、Bedrock、OpenSearch が必要になった時点で部分採用を検討 |

## 現時点の推奨

初期開発では、Tauri、React、TypeScript、SQLite、Supabase を中心に進める。まずは macOS で毎日使える品質を目指し、同期は小さく堅く始める。

この構成なら、ローカルの快適さ、クラウド同期、低コスト運用、将来の Web / iPhone 展開のバランスを取りやすい。
